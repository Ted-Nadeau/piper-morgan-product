## ADR-053 Review

### Document Summary

ADR-053 specifies the trust computation architecture implementing PDR-002's Trust Gradient Model. Key elements:

| Element | Specification |
|---------|---------------|
| **Stages** | NEW (1) → BUILDING (2) → ESTABLISHED (3) → TRUSTED (4) |
| **Progression** | 10 successful → Stage 2; 50 successful → Stage 3; explicit signal → Stage 4 |
| **Regression** | 3 consecutive negatives → drop one stage; 90 days inactivity → drop one stage |
| **Floor** | Never below Stage 2 once earned |
| **Discussability** | TrustExplainer service explains trust-based behavior when asked |

### PPM Assessment

**Approve with recommendations.**

#### Strengths

1. **"Invisible computation, visible effects"** — Correctly preserved. No trust badges, just behavior changes.

2. **Discussability built-in** — TrustExplainer service means Piper can explain herself. Critical for colleague framing.

3. **Mirrors human relationships** — Incremental building, rapid loss possible, floor prevents complete reset. This feels right.

4. **Alternatives well-reasoned** — User-controlled settings, session-only trust, ML-based scoring all rejected with clear rationale.

5. **Testable** — Pure computation logic is easy to unit test. Good architectural choice.

#### Open Questions — PPM Input

| Question | PPM Recommendation |
|----------|-------------------|
| **1. Outcome classification** | Tie to observable action, not sentiment. "User acted on response" = successful. Define per intent type. Start simple, calibrate. |
| **2. Stage 3→4 signal** | Explicit opt-in is correct. "You can just do that" or similar natural language. Don't add a settings toggle—that violates "invisible computation." |
| **3. Cross-device** | Yes, trust is per-user. No device-specific tracking needed. |
| **4. Privacy mode** | Correct to exclude. Privacy mode = no memory, no trust accumulation. |

#### Minor Recommendations

1. **Threshold calibration note**: 10 and 50 are starting points. Add explicit note that these will be calibrated based on alpha testing. Don't treat as permanent.

2. **"Successful" default**: For MVP, bias toward "neutral" when uncertain. Better to progress slowly than to inflate trust on ambiguous outcomes.

3. **Complaint detection**: "Explicit complaint" is mentioned but not defined. Consider: keywords ("stop doing that", "don't", "I didn't ask"), negative sentiment + trust-related topic, explicit "no" to proactive offer.

4. **Stage 4 reversibility**: Ensure Stage 4 → Stage 3 regression is possible (not just Stage 4 → Stage 2 on complaint). User might want less proactivity without full reset.

### Verdict

**APPROVED** — Ready for implementation with open questions resolved per recommendations above.
