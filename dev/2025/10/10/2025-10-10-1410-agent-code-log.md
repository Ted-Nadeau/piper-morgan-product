## Architecture Design

### Service Structure

```
services/briefing/
├── __init__.py              # Module exports
├── briefing_service.py      # Main service class
├── models.py                # Briefing data models
├── queries.py               # Serena query functions
└── formatters.py            # Markdown formatting
```

### Data Flow

```
1. Agent requests briefing → BriefingService.get_briefing(role)
2. Service queries Serena → queries.py functions
3. Results filtered by role → role-specific logic
4. Formatted as markdown → formatters.py
5. Return Briefing object → with metadata
```

## 6:20 PM - Post-Implementation: Preventing Double-Briefing

**User question:** "Is there anything in the existing instruction set that needs to be deprecated to avoid double-briefing the agents?"

**Investigation:** Identified three locations where agents are told to read BRIEFING-CURRENT-STATE.md for "current system state" which would conflict with Serena queries:

2. **CLAUDE.md line 209** - Footer with wrong path
3. **.cursor/rules/programmer-briefing.mdc lines 24-25** - Mandatory briefing section

**Problem:** Without updates, agents would:
- Read BRIEFING-CURRENT-STATE.md (~1,034 tokens) for system state
- Then potentially also run Serena queries (~212 tokens)
- Result: Negates our 79% token savings!
