```python
elif intent.category == IntentCategory.QUERY:
    # GREAT-4F: Smart fallback for likely mis-classified canonical intents
    text_lower = (intent.context or {}).get("original_message", intent.action).lower()

    # TEMPORAL patterns: calendar, schedule, time-related
    temporal_patterns = [
        'calendar', 'schedule', 'meeting', 'appointment',
        'today', 'tomorrow', 'yesterday', 'next week', 'this week',
        'what time', 'when is', 'what day', 'what date'
    ]

    # STATUS patterns: work status, standup, current tasks
    status_patterns = [
        'status', 'standup', 'working on', 'current',
        'progress', 'what am i', "what's my", 'my work'
    ]

    # PRIORITY patterns: importance, focus, priorities
    priority_patterns = [
        'priority', 'priorities', 'important', 'focus',
        'urgent', 'critical', 'top', 'most important'
    ]

    # Pattern matching with logging
    if any(pattern in text_lower for pattern in temporal_patterns):
        logger.warning(
            f"QUERY intent likely mis-classified TEMPORAL: {text_lower[:50]}",
            extra={"intent_id": intent.id, "category": "QUERY", "likely": "TEMPORAL"}
        )
        workflow_type = WorkflowType.GENERATE_REPORT

    elif any(pattern in text_lower for pattern in status_patterns):
        logger.warning(
            f"QUERY intent likely mis-classified STATUS: {text_lower[:50]}",
            extra={"intent_id": intent.id, "category": "QUERY", "likely": "STATUS"}
        )
        workflow_type = WorkflowType.GENERATE_REPORT

    elif any(pattern in text_lower for pattern in priority_patterns):
        logger.warning(
            f"QUERY intent likely mis-classified PRIORITY: {text_lower[:50]}",
            extra={"intent_id": intent.id, "category": "QUERY", "likely": "PRIORITY"}
        )
        workflow_type = WorkflowType.GENERATE_REPORT

    else:
        # True generic query
        logger.info(
            f"QUERY intent routed to GENERATE_REPORT: {text_lower[:50]}",
            extra={"intent_id": intent.id, "category": "QUERY"}
        )
        workflow_type = WorkflowType.GENERATE_REPORT
```

### Why These Changes Are Critical

**Production Reliability**:
- Load balancers use `/health` to route traffic
- Kubernetes uses health checks for pod readiness
- Monitoring systems use health checks for alerts
- If `/health` returns 404, service is marked unhealthy

**CI/CD Safety**:
- Strict tests act as deployment gates
- Catch regressions early in development
- Prevent broken deployments
- Fail fast with clear error messages

**Code Documentation**:
- Strict assertions communicate intent
- Clear error messages explain requirements
- No ambiguity about endpoint criticality

### The Permissive Test Anti-Pattern

**Pattern**: Tests accept both success (200) and failure (404) as valid

**Why it happens**:
- Defensive coding during development
- "Work in progress" testing approach
- Tests written before endpoint exists

**Why it's dangerous**:
- Provides false confidence
- Endpoint can be deleted without test failures
- Critical infrastructure becomes "optional"
- Production issues hidden from CI/CD

**Solution**:
1. Start with permissive tests during development
2. **TIGHTEN assertions when feature is complete** ← Critical step often missed
3. Add comments explaining why strict
4. Link to production requirements

### Decision Criteria: When to Accept 404

**NEVER accept 404 for**:
- Health checks (`/health`, `/healthz`, `/ready`)
- Core API endpoints (`/api/v1/intent`)
- Authentication endpoints
- Critical business logic

**CAN accept 404 for**:
- Optional documentation (`/docs`, `/redoc`)
- Feature flags or A/B test endpoints
- Deprecated endpoints being phased out
- Admin-only features that may be disabled

**When in doubt**: Require strict 200 and document why endpoint is critical.
