**Key Findings from ADR-034 (Plugin Architecture)**:

> "We will implement a plugin architecture where all external integrations (GitHub, Notion, Slack, Jira, etc.) are pluggable components implementing a common interface."

**Plugin Architecture Principles** (ADR-034:12-18):
1. **Domain-first**: PM concepts (WorkItem, TeamMember) drive the interface
2. **Tool-agnostic**: No tool-specific logic in core
3. **Event-driven**: Every plugin interaction emits learnable events
4. **Capability-based**: Plugins declare their capabilities
5. **Spatial-aware**: Leverage MCP spatial intelligence patterns

**Specified Plugin Structure** (ADR-034:20-28):
```
Core System
    ↓
Plugin Interface (PiperPlugin)
    ↓
Plugin Registry
    ↓
Individual Plugins (GitHub, Notion, Slack, etc.)
```

**Implementation Timeline** (ADR-034:52-76):
- Phase 1: Extract GitHub to plugin (Week 1)
- Phase 2: Plugin Infrastructure (Week 2) - Registry, discovery, config, lifecycle
- Phase 3: Second Plugin - Notion (Week 3)
- Phase 4: Spatial Integration (Week 4)

**Key Findings from ADR-013 (MCP + Spatial Intelligence)**:

> "ALL external tool integrations MUST use the unified MCP + Spatial Intelligence pattern" (ADR-013:25)

**Mandatory Pattern** (ADR-013:27-31):
1. MCP Protocol Layer: Standardized communication
2. Spatial Intelligence Layer: 8-dimensional cognitive understanding
3. No Direct API Integrations: All routed through MCP+Spatial
4. Architectural Signature: Spatial intelligence as differentiator

**8-Dimensional Spatial Context** (from ADR-013 examples):
- territory_id: Integration identifier
- room_id: Container (repo, channel, database)
- path_id: Specific item (issue, message, page)
- attention_level: Urgency/priority
- emotional_valence: Sentiment/nature
- navigation_intent: Action mode (respond, investigate, monitor)
- external_system: System name
- external_id: System-specific ID
- external_context: Additional metadata

**4. What would a plugin contract need to include?**

**Required Plugin Interface** (based on ADR-034 + current router analysis):

```python
class PiperPlugin(ABC):
    """
    Base plugin interface for all external integration plugins.

    Principles (ADR-034):
    - Domain-first: PM concepts drive interface
    - Tool-agnostic: No tool-specific logic in core
    - Event-driven: Every interaction emits learnable events
    - Capability-based: Plugins declare capabilities
    - Spatial-aware: Leverage spatial intelligence
    """

    # ===== REQUIRED: Plugin Metadata =====
    @property
    @abstractmethod
    def name(self) -> str:
        """Plugin identifier (e.g., 'github', 'slack', 'notion')"""
        pass

    @property
    @abstractmethod
    def version(self) -> str:
        """Plugin version (semver)"""
        pass

    @property
    @abstractmethod
    def description(self) -> str:
        """Human-readable plugin description"""
        pass

    @property
    @abstractmethod
    def capabilities(self) -> List[str]:
        """Declared plugin capabilities (e.g., ['issues', 'pr', 'spatial'])"""
        pass

    # ===== REQUIRED: Lifecycle Management =====
    @abstractmethod
    async def initialize(self) -> None:
        """Initialize plugin (connections, resources, etc.)"""
        pass

    @abstractmethod
    async def shutdown(self) -> None:
        """Cleanup plugin resources"""
        pass

    @abstractmethod
    def is_configured(self) -> bool:
        """Check if plugin has valid configuration"""
        pass

    @abstractmethod
    def get_integration_status(self) -> Dict[str, Any]:
        """Get plugin health/status information"""
        pass

    # ===== REQUIRED: Spatial Intelligence =====
    @abstractmethod
    def map_to_position(self, external_id: str, context: Dict[str, Any]) -> SpatialPosition:
        """Map external system identifier to spatial position"""
        pass

    @abstractmethod
    def map_from_position(self, position: SpatialPosition) -> Optional[str]:
        """Map spatial position back to external system identifier"""
        pass

    @abstractmethod
    def store_mapping(self, external_id: str, position: SpatialPosition) -> bool:
        """Persist ID mapping"""
        pass

    @abstractmethod
    def get_context(self, external_id: str) -> Optional[SpatialContext]:
        """Retrieve spatial context for external ID"""
        pass

    @abstractmethod
    def get_mapping_stats(self) -> Dict[str, Any]:
        """Get mapping statistics"""
        pass

    # ===== OPTIONAL: Domain Operations =====
    # Plugins can add domain-specific methods as needed
    # These are discovered via capability declarations
```

**5. How would plugins be discovered/loaded?**

**Plugin Registry Design** (based on ADR-034 + analysis):

```python
class PluginRegistry:
    """
    Central registry for plugin discovery, registration, and lifecycle.

    Responsibilities:
    - Plugin discovery (manual registration initially, auto-discovery later)
    - Capability indexing
    - Lifecycle management (initialize, shutdown)
    - Health monitoring
    """

    def __init__(self):
        self._plugins: Dict[str, PiperPlugin] = {}
        self._capabilities: Dict[str, List[str]] = {}  # capability → plugin names

    def register(self, plugin: PiperPlugin) -> None:
        """Register a plugin"""
        pass

    def get_plugin(self, name: str) -> Optional[PiperPlugin]:
        """Get plugin by name"""
        pass

    def find_by_capability(self, capability: str) -> List[PiperPlugin]:
        """Find plugins with specific capability"""
        pass

    async def initialize_all(self) -> None:
        """Initialize all registered plugins"""
        pass

    async def shutdown_all(self) -> None:
        """Shutdown all plugins gracefully"""
        pass

    def get_all_status(self) -> Dict[str, Dict[str, Any]]:
        """Get status of all plugins"""
        pass
```
