### 11:10 AM - Task 5: ADR-038 Decision Framework Applied to Calendar

**Reading ADR-038**: `docs/internal/architecture/current/adrs/adr-038-spatial-intelligence-patterns.md`

**Pattern Selection Criteria from ADR-038:**

1. **Domain Complexity**: Many distinct capabilities (>5) → Granular, Few (≤5) → Embedded
2. **Requirement Stability**: Evolving → Granular, Stable → Embedded
3. **Performance Criticality**: Critical → Embedded, Adequate → Either
4. **Testing Requirements**: Component isolation → Granular, Integration testing → Embedded
5. **Team Structure**: Multiple devs → Granular, Small team → Embedded
6. **Domain Nature**: Reactive coordination → Granular, Analytical → Embedded

**Calendar Analysis Against Criteria:**

| Criterion | Calendar Reality | Suggests |
|-----------|------------------|----------|
| Domain Complexity | 7 calendar + 5 spatial = 12 methods | Granular (>5) |
| Requirement Stability | Stable temporal domain | Embedded |
| Performance | Adequate (not critical) | Either |
| Testing | Would benefit from component tests | Granular |
| Team Structure | Single developer likely | Embedded |
| Domain Nature | Temporal awareness (mixed) | Neither clearly |

**DISCOVERY: Third Spatial Pattern!**

Calendar implements **neither** Granular nor Embedded patterns documented in ADR-038.

**Calendar's "Delegated MCP Pattern":**

**Structure:**
```
services/integrations/calendar/
  calendar_integration_router.py (397 lines) - Minimal router

services/mcp/consumer/
  google_calendar_adapter.py (499 lines) - MCP adapter with spatial
```

**Characteristics:**
- Router is MINIMAL (397 lines, mostly delegation)
- Spatial intelligence in EXTERNAL MCP adapter
- MCP adapter inherits from BaseSpatialAdapter
- Router delegates to MCP adapter for ALL operations
- Spatial methods available via router pass-through

**Pattern Comparison:**

| Pattern | Location | Files | Inheritance | Access |
|---------|----------|-------|-------------|--------|
| Slack (Granular) | `services/integrations/slack/` | 11 | BaseSpatialAdapter | `router.get_spatial_adapter()` |
| Notion (Embedded) | `services/intelligence/spatial/` | 1 | Standalone | `NotionSpatialIntelligence()` |
| **Calendar (Delegated)** | **Split: router + MCP** | **2** | **BaseSpatialAdapter (in MCP)** | **`router.method()` delegates** |

**Key Distinction:**
- Slack: Spatial IN integration directory
- Notion: Spatial IN intelligence directory
- **Calendar: Spatial IN MCP consumer directory**

**Why This Pattern?**

Calendar uses MCP (Model Context Protocol) for external service integration:
- GoogleCalendarMCPAdapter is MCP-based
- Follows MCP protocol patterns
- Inherits from BaseSpatialAdapter
- Router acts as thin wrapper/facade
- All implementation in MCP layer

**ADR-038 Update Needed:**

ADR-038 documents TWO patterns (Granular, Embedded) but Calendar reveals a THIRD pattern (Delegated MCP). This pattern should be documented as a valid option for integrations using Model Context Protocol.

**Recommendation**: Create ADR-039 or update ADR-038 to document "Delegated MCP Pattern" as third spatial architecture option.
