## Decisions Made

### Decision 1: HTTP 422 for Ethics Violations
**Rationale**: Ethics violations are validation errors (similar to schema validation)
**Alternative Considered**: HTTP 403 (Forbidden) or 400 (Bad Request)
**Chosen**: HTTP 422 (Unprocessable Entity) - standard for validation failures
**Outcome**: Consistent with REST semantics, proper error categorization

### Decision 2: Enable Ethics Immediately
**Rationale**: Zero users = zero risk, no gradual rollout benefit
**Alternative Considered**: Gradual rollout (Day 1 disabled, Day 2-3 enabled, Day 4+ standard)
**Chosen**: Enable immediately with feature flag (instant disable if needed)
**Outcome**: Simpler operations, ready for first user, instant rollback available

### Decision 3: Keep Current Configuration
**Rationale**: 100% test accuracy (0 false positives, 0 false negatives)
**Alternative Considered**: Tune threshold from 0.5 to higher/lower values
**Chosen**: Keep 0.5 threshold (provides 50% safety margin)
**Outcome**: No tuning needed, production-ready configuration

## Lessons Learned

### Lesson 1: PM's Role as "Simplicity Noticer"
**What Happened**: PM questioned gradual rollout necessity with zero users
**Insight**: Standard practices should be questioned when context changes
**Impact**: Eliminated unnecessary complexity (no phased rollout needed)
**Takeaway**: Don't apply standard patterns blindly - consider actual context

### Lesson 2: HTTP Status Codes Matter
**What Happened**: Test script initially expected wrong status codes
**Insight**: Ethics violations are validation errors (422), not authorization failures (403)
**Impact**: Tests showed false failures until status codes corrected
**Takeaway**: Use semantically correct HTTP status codes for proper error categorization

**Insight**: Documentation review found zero issues, all production-ready
