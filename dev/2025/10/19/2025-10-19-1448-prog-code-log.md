## Architecture Decisions

### 1. **Mode Naming Consistency**
- **API**: Uses simple names (standard, issues, documents, calendar, trifecta)
- **Service**: Uses descriptive names (standard, with_issues, with_documents, with_calendar, trifecta)
- **Mapping**: Mode mapping dictionary in endpoint to translate API → Service

**Rationale**: User-friendly API while maintaining service clarity

### 2. **Format Converters**
- **Location**: Inline in standup.py rather than separate module
- **Approach**: Simple formatters for slack/markdown/text

**Rationale**:
- Phase 2 scope - simple implementation
- Easy to extract to module later if needed
- Keeps all standup API logic in one file

### 3. **Authentication**
- **Approach**: Optional X-API-Key header
- **Implementation**: Placeholder dependency (returns None for Phase 2)
- **Fallback**: user_id from request body or config resolution

**Rationale**:
- Phase 2 focus on core functionality
- Auth enhancement deferred to future phase
- Existing app.state patterns used

### 4. **Error Handling**
- **Pattern**: Pattern-034 (REST-compliant error responses)
- **Implementation**: Via `web/utils/error_responses.py`
- **Coverage**: validation_error (422), internal_error (500)

**Rationale**:
- Follows existing codebase patterns
- Consistent with other endpoints
- User-friendly error messages

### 5. **Dependency Injection**
- **Approach**: FastAPI dependencies via Depends()
- **Service Access**: Via app.state.service_container
- **Service Creation**: Direct instantiation (StandupOrchestrationService())

**Rationale**:
- Follows FastAPI best practices
- Integrates with existing ServiceContainer
- Easy to test with mocks

### 6. **OpenAPI Documentation**
- **Approach**: Auto-generated via FastAPI/Pydantic
- **Models**: Comprehensive Pydantic models with examples
- **Access**: /docs endpoint (Swagger UI)

**Rationale**:
- Zero maintenance cost
