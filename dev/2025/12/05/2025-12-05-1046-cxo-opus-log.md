### 10:48 AM - Wardley Component Calibration

Addressed PM's question: What does Wardley consider a "component" vs how our codebase defines itself?

**Key insight**: Wardley components ≠ code modules

- Multiple code modules → one Wardley component (e.g., Knowledge Graph + Semantic Indexing = "Contextual Awareness")
- One concept → no discrete code module (e.g., "Trust Architecture" is emergent behavior)
- Infrastructure enables but isn't the value (Redis enables sessions, but "Session Continuity" is the component)

**Proposed calibration framework:**

| Level | What It Is | On Map? |
|-------|-----------|---------|
| User Need | What PM actually wants | Yes (top) |
| Capability | What Piper can do | Yes |
| Mechanism | How capability works | Maybe (if novel) |
| Implementation | Code that realizes it | Rarely |
| Infrastructure | What code runs on | Yes (commodity anchor) |

**The test for any component:**
1. Would a competitor need to replicate this to match our value?
2. Is this product language or code language?
3. Does this serve users directly or enable something that does?

**Proposed simplified map structure:**

```
USER NEED

CAPABILITIES (Genesis/Custom)
```
