## Architecture Changes

### Before Refactoring:
```
todos table (standalone)
├── id, title, description, priority, status
├── parent_id, position, due_date, ...
├── created_at, updated_at, completed_at
└── 30+ fields total
```

### After Refactoring:
```
items table (base for all item types)
├── id, text, position, list_id
├── item_type (discriminator: 'item', 'todo', etc.)
├── created_at, updated_at

todo_items table (todo-specific data)
├── id (PK + FK to items.id)
├── description, priority, status, completed
├── parent_id, due_date, reminder_date, ...
├── 24 todo-specific fields
└── Foreign key to items.id

Joined via: items.id = todo_items.id WHERE item_type = 'todo'
```

### Domain Model:
```python
class Item:  # Base primitive
    id: str
    text: str          # ← Universal field
    position: int
    list_id: Optional[str]
    created_at: datetime
    updated_at: datetime

class Todo(Item):  # Extends Item
    # Inherits: id, text, position, list_id, created_at, updated_at
    # Todo-specific:
    description: str
    priority: str
    status: str
    completed: bool
    ... (24 more fields)

    @property
    def title(self):  # Backward compatibility
        return self.text
```

## Key Design Decisions (Requested by PM at 12:30 PM)

### 1. ENUM vs String Types
**Decision**: Use String types in database, not PostgreSQL ENUMs
**Rationale**:
- Phase 2 migration created VARCHAR columns
- ENUMs are rigid (require migrations to add values)
- Strings are flexible and match migration intent
- Eliminates type casting complexity

### 2. Service Inheritance Pattern
**Decision**: TodoService extends ItemService (not composition)
**Rationale**:
- Clear IS-A relationship (Todo IS an Item)
- Matches domain model inheritance (Todo extends Item)
- Matches database inheritance (TodoDB extends ItemDB)
- Enables polymorphic operations naturally

### 3. Dependency Injection
**Decision**: Services created on-demand in FastAPI dependencies
**Rationale**:
- No global state/singletons needed
- Proper async context management
- Easy to test/mock
- Follows FastAPI best practices

### 4. Repository Access
**Decision**: Services use repositories internally, not exposed to API/handlers
**Rationale**:
- Clean separation of concerns
- Services provide business logic layer
- Repositories handle data access
- Easier to change persistence layer later
