## Naming Analysis: PPM Input

### Context

The Lead Developer completed a capabilities naming analysis identifying 35+ capabilities with 5 naming patterns, creating user confusion. A 4-tier framework is proposed. The memo asks for CXO/PPM/Comms input on specific naming decisions.

### My Input on Open Questions

**1. Naming Tone Balance (Plain Language vs. Clever/Memorable)**

**Recommendation**: Default to plain language (register a), reserve clever naming (register c) for 3-5 flagship differentiators.

**Rationale**:
- Most capabilities are utility functions—"Create Issue," "Upload Document" should be invisible-obvious
- Clever names work for features that ARE Piper's differentiation: Morning Standup, portfolio onboarding, spatial awareness
- Overusing clever names dilutes impact and creates cognitive overhead
- The 10%/90% rule applies here too: 10% memorable flagship names, 90% functional clarity

**Principle I'd suggest**: "If the feature would exist in any PM tool, use plain naming. If the feature is uniquely Piper, consider distinctive naming."

**2. "Focus Assistant" and "Standup Assistant"**

**Assessment**: "Assistant" framing works well with our "colleague not tool" positioning.

- **Standup Assistant**: ✅ Clear, maps to existing "Interactive Standup" capability
- **Focus Assistant**: ⚠️ Less clear—what does "focus" mean in practice? "Priority guidance" is the underlying capability. Consider "Priority Assistant" or simply "What should I focus on?" as the natural query with no branded wrapper.

**Alternative framing**: Not everything needs a product name. Some capabilities are better surfaced as natural queries:
- "What should I focus on?" (instead of "Focus Assistant")
- "Prep for my next meeting" (instead of "Meeting Prep Assistant")

This aligns with PDR-002's discovery-through-conversation philosophy.

**3. GitHub Feature Framing (PM-Centric vs. Developer-Centric)**

**Strong preference for "Backlog" framing**.

| Option | Assessment |
|--------|------------|
| "GitHub Tools" | ❌ Too developer-centric; sounds like code tools |
| "Issue Tracker" | ⚠️ Generic; doesn't signal PM value |
| "Backlog Tools" | ✅ Agile/PM framing; clear without jargon |
| "Project Tracker" | ⚠️ Overlaps with our existing "Projects" concept |

"Backlog Tools" signals:
- PM ownership (product backlog, not code backlog)
- Agile methodology alignment
- Strategic rather than tactical framing

**4. "Don't Miss" or Similar Unique Naming**

**Assessment**: This is a candidate for the 10% flagship naming—but needs careful consideration.

**In favor**:
- "Don't Miss" is memorable and actionable
- Describes user benefit, not technical function
- Could become a signature Piper feature ("What did I miss?" → "Here's your Don't Miss list")

**Concerns**:
- May feel too casual for enterprise contexts
- "Attention items" or "What needs attention?" is clearer
- Could test with alpha testers

**Principle I'd propose**: If we use distinctive naming, it should:
1. Be immediately comprehensible without explanation
2. Describe user benefit, not system behavior
3. Be conversationally natural ("Show me my Don't Miss" works; "Show me my Smart Notifications" doesn't)
4. Pass the "would a colleague say this?" test

**5. Additional Observations**

**Category naming inconsistency**: The report proposes "Standup Tools" and "Slack Features" and "Calendar Features"—mixing "Tools" and "Features." Should standardize.

**Suggestion**: Use "Tools" for action-oriented categories (Backlog Tools, Document Tools), "Features" sounds more passive.

**Integration language**: Current naming sometimes leads with integration ("GitHub issue creation," "Notion search"). Consider flipping: "Create Issue (GitHub)" puts the action first, integration second. This matches how users think ("I want to create an issue" not "I want to do something with GitHub").
